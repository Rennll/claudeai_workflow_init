# protocol.md — 工作協議

## Session 開場流程
每個 session 開始時依序讀取：
1. `project.md`（專案方向與原則）
2. `handoff.md`（當前狀態與下一步）

確認本 session 的單一交付目標後才開始執行。

## 偏移偵測
遇到以下三類情況立即暫停，說明觀察到的偏移，問使用者要修正方向、調整藍圖、還是這本來就是計畫的一部分：
- **目標偏移**：當前子任務不再直接服務主線目標
- **成本偏移**：某個工具或格式問題佔據比主線任務更多的討論篇幅
- **證明偏移**：開始追求「讓某技術細節能運作」而非完成功能

## Wrap-up 規則
- 任務到達自然收束點時主動建議 wrap up，使用者確認後才產出輸出
- 每個 session 結束必須產出完整 `handoff.md` 與 commit 指令
- `handoff.md` 內的 session 歷史只保留最近兩筆，舊的壓縮後移除
- 藍圖方向改變時同步更新 `project.md` 並記錄原因與版本

## 狀態確認機制
遇到以下情況，在回答前先說「⚠️ 狀態確認：xxx」再繼續：
- 回報 bug 但未說明是否使用最新版本——確認後直接從其他方向排查，不重複確認
- 同時提出兩個以上不同性質的問題
- 同一問題修超過兩輪未解決——停下來說明可能的根本原因，確認方向後再繼續

## 互動偏好
- 輸出程式碼時預設用程式碼區塊。僅在需要於容器內執行或測試時才建立檔案，並說明原因。
- commit message 由 AI 根據本次 log 產出
