**寫 CLAUDE.md 前問自己：這件事 agent 自己讀程式碼會發現嗎？會的話不用寫。**

- 目標 300 行以內，能 60 行更好
- 每一行都吃上下文配額，要值得
- `/init` 出來的不要直接用，平均降 3% 成功率

**該寫的：**
- 非預期的工具鏈選擇（用 `uv` 不是 `pip`、用 `bun` 不是 `npm`）
- 歷史包袱、踩過的雷、團隊共識
- 決策邏輯和行為偏好（不確定時選 A 還是 B）
- 硬邊界（絕對不能碰的路徑、絕對不能做的操作）
- 非預期的衝突處理方式
- 代碼範例 > 長篇解釋——一個示範你 codebase 風格的真實片段，勝過幾段抽象描述。

**不該寫的：**
- 目錄結構（`ls` 就能看到）
- README 和文件裡已經有的內容
- 技術棧裡顯而易見的細節（`package.json` 寫了就不用再說）
- 程式風格規定（交給 linter）
- 空泛的標語（「寫 clean code」、「performance matters」）
- 只在特定情境才用的指令（拆到子檔案）

**超過 150 行就考慮分層：**
- 主檔放路標，細節用 `@import` 拉進來
- 子目錄各自一份，agent 會階層式合併
- 多工具團隊用 symlink，不要維護平行版本

**真正的紅線不要只靠 CLAUDE.md：**
- CLAUDE.md 順從率約 70%，硬規則要用 hooks 寫
- `PreToolUse` 回 exit code 2 才是真的攔得住

**心態：**
- 你不是在寫 README，是在設計一個每次都失憶的 agent 開始工作前的記憶植入
- 寫偏好比寫事實有價值，結構 agent 自己摸得到，行為偏好它摸不到
- 文件過期 agent 就會出錯，要納入維護週期
